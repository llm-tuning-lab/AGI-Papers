코딩 에이전트의 역설: 친절한 설명서 (AGENTS.md)가 망치는 효율성

우리는 최근 코딩 에이전트 도구(예: Claude Code, OpenClaw 등)를 쓸 때, 리포지토리 최상단에 `AGENTS.md`나 `README.md` 같은 두꺼운 지침서 파일을 던져주곤 합니다. 주요 AI 기업들도 "배경지식을 주면 에이전트가 코딩을 더 잘할 것"이라고 권장하기 때문이죠.

하지만 최근 발표된 논문 **"Evaluating AGENTS.md: Are Repository-Level Context Files Helpful for Coding Agents?"**는 놀라운 반전을 보여줍니다. "주인님이 주신 이 친절한 파일이 사실 AI를 더 소심하고 멍청하게 만들고, 당신의 지갑을 축내고 있다"는 충격적인 팩트 체크를 4가지로 요약합니다.

1. "지침서를 지키려다 본질을 놓친다" (성능 저하)
12개 리포지토리의 138개 복잡한 버그 수정/기능 추가 벤치마크(AGENTBENCH) 테스트 결과:
- LLM이 정성스레 자동 생성한 컨텍스트 파일을 주면, 아무 파일도 안 줬을 때보다 작업 성공률이 오히려 **평균 3% 떨어졌습니다.** 
- (심지어 사람이 직접 손으로 깎은 컨텍스트를 주더라도 성능 향상은 겨우 평균 4% 상승에 그쳤습니다.)

2. 배보다 배꼽이 더 큰 연산 비용 (정보 과부하)
문제가 생깁니다. 컨텍스트 파일에 적힌 "A 파일은 이런 역할을 하고 테스트는 B 규칙을 따르라"는 문구에 에이전트가 지나치게 목을 매기 시작합니다.
- 불필요하게 스크립트 탐색 범위를 넓히고, 코드를 쓸데없이 더 많이 읽고 쓰면서 작업 스테핑(Steps)과 추론 비용(API Cost)이 **20% 이상 폭등**해버립니다.

3. 요약본은 지름길이 되지 않는다
우리는 AGENTS.md가 리포지토리 지도 역할을 해서 에이전트가 정답 파일로 바로 다이빙할 줄 알았습니다. 
하지만 현실 로그(Trace analysis)를 뜯어보니 에이전트는 이미 다른 README를 알아서 찾아 읽고 있었습니다. 중복된 문서들 속에서 쓸데없는 생각(Reasoning) 토큰만 길어지며 작업 난이도만 증폭되는 부작용이 속출했습니다.

4. 진실: 그냥 스스로 부딪히고 깨지게 두어라
결론은 명쾌합니다. 거대한 가이드라인을 강제하기보다 "차라리 에이전트가 오류와 함께 직접 한두 번 굴러보게 내버려 두는 것"이 훨씬 효율적이고 저렴하고 정확하다는 것입니다.

💡 마치며: B2B 파이프라인의 설계 지침
이 연구는 소형 에이전트 모델이나 로컬 최적화 파이프라인을 구축하는 설계자들에게 확실한 인사이트를 줍니다. 
무거운 룰 기반의 프롬프트나 지침서를 덕지덕지 바르면 오버헤드만 커집니다. 차라리 "꼭 필요한 도구 사용법" 등 핵심 1~2줄만 선언하고 과감하게 에이전트 스스로의 검색(GraphSearch) 추론에 파이프라인을 온전히 맡기십시오. 

완벽한 매뉴얼보다, 자유로운 실행(Execution)이 AGI를 더 똑똑하게 만듭니다.
